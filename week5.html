OS Fingerprinting

Identifying remote operating systems using Nmap active scanning and p0f passive analysis
Kali Linux 2025 Nmap 7.x p0f v3 Active + Passive Updated — xprobe2 Removed

02 Key Concepts
OS Fingerprinting
Determining which operating system runs on a remote host by analysing how its TCP/IP stack behaves — without logging in.
TCP/IP Stack
Each OS implements TCP/IP slightly differently — TTL defaults, window sizes, option ordering, SYN response flags. These differences are the fingerprint.
TTL (Time-To-Live)
A field in IP headers. Windows defaults to 128, Linux to 64, Cisco to 255. A blunt but fast OS indicator.
TCP Window Size
The buffer size advertised in SYN packets. Different OSes use characteristic default values that can aid identification.
Active Fingerprinting
Sends crafted probes to the target and analyses responses. Fast and accurate — but detectable by IDS/IPS. Tool: Nmap -O.
Passive Fingerprinting
Listens silently to existing traffic. No packets sent to the target — nearly undetectable. Depends on observing live connections. Tool: p0f.
Promiscuous Mode
Forces the NIC to capture all packets on the segment, not just those addressed to your machine. Required for p0f to observe other hosts' traffic.
Ephemeral Port
Temporary port (49152–65535) auto-assigned by the OS for outbound connections. Visible in p0f output alongside the fixed server port.
03 Active vs Passive — What Changed in 2025

The two fundamental approaches to fingerprinting have not changed. The tools we use have.

Aspect Active Fingerprinting Passive Fingerprinting
How it works Sends crafted TCP/UDP/ICMP probes; analyses responses Listens to existing traffic; no packets sent to target
Speed Fast — probes get immediate responses Depends on observing live traffic
Accuracy High — deliberate probing gives clear signatures Good — but depends on traffic quality
Detectability Easily detected by IDS/IPS Near-undetectable — read-only

New tool (2025) Nmap (-O) — industry standard p0f v3 — updated and active

04 Pre-Lab Setup

Complete all four steps before starting the activities. Do not skip the interface check — this is the most common source of errors in Kali 2025.

  • 1
    Find your network interface name

    Kali 2025 uses predictable interface names. eth0 may not exist. Always check first.

    kali — check interface
    $ ip a 1: lo: <LOOPBACK> ... inet 127.0.0.1/8 2: enp0s3: <BROADCAST,MULTICAST,UP> ... link/ether 08:00:27:xx:xx:xx inet 10.0.2.15/24 brd 10.0.2.255 scope global dynamic enp0s3

    In this example the active interface is enp0s3. In VirtualBox it may be eth0; in VMware commonly ens33. Note the name — you will need it in Activity 2.

  • 2
    Find the target IP address

    On the target Windows machine, open Command Prompt and run:

    windows — target machine
    C:\> ipconfig Windows IP Configuration Ethernet adapter Ethernet: IPv4 Address. . . . . : 10.0.2.28

    Note this IP. You will use it as <TARGET_IP> throughout this workshop.

  • 3
    Confirm Nmap is installed
    kali
    $ nmap --version Nmap version 7.94 ( https://nmap.org ) # If missing: sudo apt update && sudo apt install nmap -y
  • 4
    Confirm p0f is installed
    kali
    $ p0f -h --- p0f 3.09b by Michal Zalewski --- # If missing: sudo apt update && sudo apt install p0f -y
05 Activity 1 — Active OS Fingerprinting with Nmap

Nmap is the most widely used network scanner in the world. Its OS detection engine sends a carefully crafted series of TCP, UDP, and ICMP probes to the target and compares the responses against a database of thousands of known OS signatures. This directly replaces xprobe2 from the original workshop — and is significantly more accurate.

Objective: Use Nmap to actively identify the operating system of the target Windows host. Work through Parts A–D in order, observing how additional flags change the depth of information returned.

PART A — Basic Port Scan

Start with a standard port scan. This generates no OS guess yet — it simply confirms reachability and identifies open ports, which Nmap will need for OS detection in the next step.

  • 1
    Open a terminal and run the basic scan
    equivalent to the original xprobe2 -T 1-1024 scan
    $ nmap -p 1-1024 <TARGET_IP> # Example output: PORT STATE SERVICE 80/tcp open http 135/tcp open msrpc 139/tcp open netbios-ssn 445/tcp open microsoft-ds
  • 2
    Note the open ports

    Common open ports on a Windows host: 80 HTTP, 135 RPC, 139/445 SMB, 3389 RDP. At least one open and one closed port are required for reliable OS detection in Part B.

PART B — OS Detection with -O

The -O flag enables Nmap's OS detection engine. Raw socket access is required, so you must use sudo.

  • 1
    Run the OS detection scan
    $ sudo nmap -O <TARGET_IP>
  • 2
    Examine the OS detection section of the output
    nmap -O output (example)
    Device type: general purpose Running: Microsoft Windows 10|11 OS CPE: cpe:/o:microsoft:windows_10 OS details: Microsoft Windows 10 1903 - 21H2 Network Distance: 1 hop
  • 3
    Record the result

    Note the OS name, CPE identifier, and the version range reported. If Nmap returns multiple candidates with a percentage confidence, record the top result and its percentage.

Why sudo? OS fingerprinting requires raw socket access to craft and send custom TCP probes. The kernel restricts this to root. Always run Nmap with sudo when using -O, or the OS detection module will be silently skipped.

PART C — Forced Guess with --osscan-guess

When Nmap cannot make a confident determination — typically because too few open ports are available — it may refuse to print a result. The --osscan-guess flag forces it to print its best guess anyway, along with a confidence percentage.

  • 1
    Run the forced-guess scan
    $ sudo nmap -O --osscan-guess <TARGET_IP>
  • 2
    Compare with Part B

    Does the guess change when confidence is lower? Does the reported OS version range widen? Record your observations — this will feed into the review questions.

PART D — Combined OS + Service Detection (Recommended)

In a real penetration test, OS detection is almost always run alongside service version detection. The -sV flag probes open ports to identify the software running on each one. Together, -sV -O gives a far more complete picture of the target.

  • 1
    Run the combined scan
    $ sudo nmap -sV -O <TARGET_IP>
  • 2
    Note the Service Info line

    At the bottom of the output, the Service Info line shows OS hints derived from service banners. These can supplement or confirm the OS detection result from -O.

  • 3
    Optional — run the aggressive mode scan
    $ sudo nmap -A <TARGET_IP> # -A is shorthand for: -O + -sV + -sC (default scripts) + --traceroute # Most complete single-command scan — but also the noisiest and most detectable
06 Activity 2 — Passive OS Fingerprinting with p0f

p0f (Passive OS Fingerprinting) is a read-only network tool — it never sends any packets. It listens on a network interface and analyses the characteristics of packets that pass by, in particular the TCP SYN used to establish connections. From these it infers the remote OS, hop distance, and connection type.

p0f version 3 is a complete rewrite of the original codebase with significantly improved network-level fingerprinting and support for reasoning about application-layer payloads.

Objective: Capture and analyse TCP/IP traffic passively. Identify the operating system of other hosts communicating with your machine — without sending a single packet to the target.

PART A — Review Available Options

  • 1
    Print the p0f help screen
    $ p0f -h

The key flags used in this workshop:

FlagPurpose
-hDisplay help and full list of options
-LList all available network interfaces for packet capture
-i <interface>Specify which interface to listen on — e.g. -i eth0 or -i enp0s3
-pEnable promiscuous mode — capture all packets on the network, not just those addressed to your machine
-o <file>Write all output to a log file for later review

PART B — List Network Interfaces

  • 1
    Run the interface listing command
    kali — list capture interfaces
    $ p0f -L 0: lo (no description) 1: enp0s3 (no description) 2: any Pseudo-device that captures on all interfaces
  • 2
    Identify your active interface

    Use the same interface name you confirmed in Pre-Lab Step 1. Do not use lo (loopback — will only see traffic to/from your own machine). Do not blindly type eth0.

Most common mistake: typing sudo p0f -i eth0 -p when the interface is named enp0s3. p0f will fail immediately with a "No such device" error. Always verify with p0f -L or ip a first.

PART C — Start p0f in Promiscuous Mode

  • 1
    Start p0f — replace <INTERFACE> with your interface name from Part B
    $ sudo p0f -i <INTERFACE> -p # Example — VirtualBox (most common): $ sudo p0f -i enp0s3 -p # Example — VMware: $ sudo p0f -i ens33 -p
  • 2
    p0f starts — but shows no output yet
    kali — p0f listening
    --- p0f 3.09b by Michal Zalewski --- [+] Closed 1 file descriptor. [+] Listening on interface 'enp0s3'. [+] Default packet filter applied. Monitoring traffic. Waiting for traffic...
Why no output? p0f only produces results when it observes a TCP connection being established. It is waiting for a SYN packet. You need to generate traffic in the next step for anything to appear.

PART D — Generate Traffic and Observe Results

With p0f running in one terminal, open a second terminal or the Firefox browser on Kali to generate TCP connections.

  • 1
    Open Firefox on Kali and visit any HTTP/HTTPS address

    For example, your router's management console, a local web server, or any external site:

    # Try your router admin page (if accessible on the lab network) http://192.168.x.x # Or visit a local web server running on the target machine http://<TARGET_IP>
  • 2
    Switch back to the p0f terminal — you should now see fingerprint results
    p0f — captured fingerprint (example)
    .-[ 192.168.1.105/54321 -> 54.70.252.70/443 (syn) ]- | | client = 192.168.1.105/54321 | os = Linux 3.11 and newer | dist = 0 | params = generic ethernet | raw_sig = 4:64+0:0:1460:mss*20,7:mss,sok,ts,nop,ws:df,id+:0 `----
  • 3
    Identify each field in the output
    FieldMeaning
    clientIP address and ephemeral port of the machine that initiated the TCP connection
    osp0f's best OS guess based on TCP SYN characteristics (TTL, window size, options)
    distEstimated hop count between p0f and the observed host. 0 = local segment
    paramsInferred connection medium (e.g. generic ethernet, DSL, modem, satellite)
    raw_sigThe raw TCP signature used for the fingerprint match — useful for manual analysis
  • 4
    Generate traffic from multiple sources

    Try creating connections from the target Windows machine, your host OS, and any other device on the lab network. Observe how p0f identifies each one differently. Compare the os field and raw_sig values across different hosts.

  • 5
    Stop p0f when done
    # Press Ctrl+C in the p0f terminal to stop capture ^C [+] Captured 14 packets total.

PART E — Optional: Save Output to a Log File

For submission or later review, redirect p0f output to a file using the -o flag:

$ sudo p0f -i enp0s3 -p -o /home/kali/p0f_results.txt # After capturing, view the log: $ cat /home/kali/p0f_results.txt
07 Discussion Questions
  • ?

    Nmap -O requires at least one open and one closed port to make a reliable OS determination. Why are both needed — what does each contribute to the fingerprinting process?

  • ?

    In Part C of Activity 1, --osscan-guess forced Nmap to print a result even when confidence was low. What are the risks of acting on a low-confidence OS fingerprint during a penetration test?

  • ?

    p0f never sends any packets to the target. What are the implications of this for an attacker conducting passive reconnaissance? What are the implications for a defender trying to detect reconnaissance activity?

  • ?

    Compare the OS identification results from Nmap -O and p0f for the same host. Were they consistent? If they differed, what might explain the discrepancy?

  • ?

    Some OSes and firewalls are specifically configured to defeat fingerprinting — by randomising TTL values, altering TCP window sizes, or changing option ordering. Name one such technique and explain how it would affect the results from both Nmap and p0f.

09 Security Implications

OS fingerprinting sits at the beginning of most attack chains. Understanding what it reveals — and how to limit that — is as important as knowing how to run the tools.

Targeted Exploitation
Knowing the exact OS version allows an attacker to select CVEs that apply specifically to that target. A Windows 10 1903 host has a very different attack surface to a Windows 11 23H2 host.
Reconnaissance Phase
Fingerprinting is a standard step in the PTES (Penetration Testing Execution Standard) and OWASP testing methodology. It precedes vulnerability scanning and exploitation.
IDS Detection
Active fingerprinting (Nmap) is easily detected by Snort, Suricata, and commercial IDS/IPS. The unusual probe packets have well-known signatures. Passive tools like p0f leave no trace.
Obfuscation
Tools like Packet Fence, PfSense rules, or custom iptables configurations can alter TCP/IP headers to mislead fingerprinting tools — a defensive countermeasure called OS personality spoofing.
Asset Inventory
Defenders use the same techniques — running Nmap -O across their own network to build an accurate OS inventory and detect unauthorised or unexpected hosts.
Patch Management
Knowing which OS versions are deployed is a prerequisite for effective patch management. If you cannot enumerate them, you cannot patch them systematically.
